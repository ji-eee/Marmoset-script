# 테스트 현황 · Marmoset 검증 체크리스트 · 알려진 한계

## 맥에서 검증한 것 (Marmoset 없이)

`projbake` 코어는 `mset` 비의존이라 로컬(파이썬 3.9)에서 전부 테스트됨.

- `tests/test_core.py` — 단위 테스트(전부 통과)
  - 벡터/행렬, `rot_y(90)`가 +X→-Z(오른손 Y-up), 강체 역행렬 왕복
  - 카메라 투영: 정면 점→화면 중앙, +X→오른쪽, +Y→위, 뒤쪽 점은 not-in-front
  - 바이리니어 샘플링, 경계 밖 None
  - PNG 왕복 + 필터 5종(None/Sub/Up/Average/Paeth) 디코드
  - 팔레트 PNG: 정상 디코드 / 인덱스 범위 초과 무크래시 / PLTE 없음 → 깔끔한 ValueError
  - 베이크: 정면 샘플, 옆면 마스킹(투명), 후면(180도 회전 후 후면 캡처 샘플), 오클루전
  - 크로스오브젝트 오클루전: 깊이 bias 안쪽으로 붙은 앞 오브젝트(깃털)를 ID 버퍼로 구분 → 뒤 표면에 안 묻음
  - edge_blur: 내부 원본 유지, 경계 페더링, 검은 헤일로 없음, radius 0 = no-op
- `tests/make_sphere_demo.py` — UV 스피어 합성 씬 라운드트립
  - 정/후면 캡처를 UV 인코딩 색으로 포워드 렌더 → 베이크 → 텍셀이 자기 UV를 복원(오차 0~1/255)
  - `baked.png`(masked), `baked_masked_blur.png`(테두리 블러), `baked_full.png`(옆면 미마스킹) 생성
- `tests/test_plugin_mock.py` — 가짜 `mset`로 플러그인 오케스트레이션 통합 테스트
  - 머리(구) + 앞을 가리는 깃털(쿼드) + 숨긴 오브젝트로 구성
  - 오브젝트별 격리 캡처(나머지 숨김) → 180도 회전 후면 캡처 → transform·가시성 정확 원복
  - **깃털 뒤 머리 영역이 채워짐(비지 않음) + 머리 색(깃털 번짐 아님)** 검증 ← 이번 핵심
  - 메테리얼별 `_masked`/`_full` 2장 + 깃털 텍스처 출력, 라운드트립, `_full` 커버리지 > `_masked`

실행:
```
python3 tests/test_core.py
python3 tests/make_sphere_demo.py [출력폴더]
python3 tests/test_plugin_mock.py
```

## 적대적 코드 리뷰 결과

코어 수학(투영/래스터/오클루전/마스킹/변환)은 리뷰에서 **결함 없음**으로 확인.
발견·수정한 것은 PNG 디코더 2건뿐(둘 다 반영 완료):
- 팔레트 PNG의 PLTE 경계 검사 누락 → 방어 코드 + 테스트 추가
- 죽은 변수/미사용 함수 제거

## 실제 Marmoset(Windows)에서 반드시 확인할 것

맥에서 검증 불가능한 부분. 아래 순서로 점검 권장.

1. 플러그인 로딩·UI 표시
   - 폴더째 복사(`CaptureFrontBackBake.py` + `projbake/`) 후 Edit > Plugins > Refresh
   - 창이 뜨고 버튼/리스트박스/텍스트필드가 정상인지. 생성자 시그니처가 다르면 로그(`mset.err`) 확인
2. 캡처(오브젝트별 격리)
   - 격리 캡처 때 나머지 오브젝트가 실제로 숨겨졌다가 원복되는지(`visible` 토글)
   - 겹친 뒤쪽 오브젝트 면이 결과에서 채워지는지(요구사항 핵심)
   - 중간 `_capture_*.png`가 베이크 후 출력 폴더에 남지 않는지(자동 삭제됨.
     디버깅으로 캡처 원본이 필요하면 `run_bake`의 `os.remove` 부분을 잠시 주석 처리)
   - `_full` 출력이 투명 영역 없이 완전 불투명인지, 모델에 적용 시 흰 테두리가 없는지
   - 요청한 정사각 해상도로 나오는지(아니어도 코드는 실제 크기로 투영 보정함)
3. **Euler 규약(가장 중요)**
   - 베이크 결과가 좌우 반전/상하 반전/틀어짐 없이 UV에 맞게 찍히는지
   - 어긋나면 `CaptureFrontBackBake.py`의 `EULER_ORDER`를 바꿔 재시도
     (후보: `"YXZ"`(기본) → `"XYZ"`, `"ZYX"`, `"YZX"` 등. 카메라와 턴테이블이 같은 값을 공유)
   - 판정 팁: 정면만 있는 단순 케이스(카메라 pitch≈0)는 어떤 순서든 같으니, 3/4 뷰로 각도 준 카메라에서 검증
4. 턴테이블 원복
   - 베이크 후 씬의 오브젝트 위치/회전이 원래대로인지(코드는 `finally` 원복이지만 실물 확인)
5. 후면 정합
   - 후면 영역 텍스처가 뒤통수 UV에 맞게 들어갔는지. 어긋나면 회전 중심(C)·`flip_v`·`EULER_ORDER` 순으로 점검
6. UV v 방향
   - 텍스처를 다시 메테리얼에 물렸을 때 상하가 맞는지. 뒤집혔으면 `bake_scene(..., flip_v=False)`

## 알려진 한계 / 향후

- 성능: numpy 없이 순수 파이썬. 오브젝트별 격리 캡처라 렌더는 오브젝트 수 × 2회.
  오브젝트가 많거나 4096 + 고폴리면 느려짐 → 1024~2048부터. 캡처는 결국 GPU 렌더라 부담은 크지 않음
- 격리 캡처는 대상 오브젝트가 최상위(top-level)일 때 정확. 다른 대상의 자식으로 들어간 경우
  숨김/회전이 부모 기준이라 어긋날 수 있음(대부분의 분리 오브젝트는 top-level)
- 노멀 없는 메시는 지오메트리 노멀로 대체. 와인딩이 반대면 정/후면이 뒤집힐 수 있음(임포트 메시는 보통 노멀 있음)
- UV 없는 메시는 베이크 불가(텍셀↔3D 매핑이 없음)
- non-identity 회전 오브젝트: `EULER_ORDER`가 실제 Marmoset과 정확히 맞아야 후면 정합이 정확.
  identity/yaw만 있는 경우(대부분의 캐릭터)는 안전
- UDIM/타일 UV 미지원(0~1 UV 가정)
- 향후: numpy 가속(삼각형 벡터화), Marmoset 깊이 패스로 오클루전 대체, UV seam 패딩 옵션
