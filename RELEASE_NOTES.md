## v1.1 — projbake 성능 최적화 (2026-07-09)

순수 파이썬 코어(`projbake`)의 핫패스를 최적화했다. **출력은 이전과 바이트 단위로 완전히 동일**하며(부동소수점 연산 순서를 그대로 보존), 실측 기준 약 **1.3~1.4배 빨라졌다**.

### 왜 순수 파이썬인가
Marmoset 내장 파이썬에는 numpy/Pillow가 없어 표준 라이브러리만으로 동작해야 한다. 따라서 "삼각형 벡터화" 같은 numpy 가속은 쓸 수 없고, 파이썬 레벨에서 호출 오버헤드·중복 계산·임시 객체 할당을 줄이는 방식으로 최적화했다.

### 변경 내용
- **`image.sample_bilinear_weighted`** — 텍셀당 호출되는 알파 가중 바이리니어 샘플링에서 `get()` 4회를 `data` 직접 인덱싱으로 인라인하고 4-샘플 누적 루프를 언롤. 연산 순서 동일.
- **`mesh.iter_world_triangles`** — 정점은 삼각형 약 6개가 공유하므로, 결정적인 정점별 월드 좌표·월드 노멀을 호출 단위로 캐싱해 `transform_point`/`v_normalize` 중복 호출 제거.
- **`bake._bake_triangle`** — 가장 뜨거운 텍셀 루프에서 `camera.project`·`v_normalize`·`v_sub`·`transform_point`·`transform_dir3`·`_visible`를 삼각형당 상수 호이스팅 + 지역 클로저로 인라인(원근/직교 경로 모두 보존). 카메라 뷰·턴테이블 행렬은 강체(rigid)라 동차 나눗셈이 1.0으로 나누는 것이므로 비트 변화 없이 생략.
- **`bake.build_depth_id`** — 깊이/ID 버퍼 빌드의 정점별 투영을 인라인하고 카메라 뒤 삼각형을 조기 스킵(미사용 `_tri_screen` 제거).
- **`postprocess`** — box-blur 프리픽스 합을 `itertools.accumulate`(C 레벨) + 스트라이드 슬라이싱으로, `_nearest_painted_map`의 알파 스캔을 `d[3::4]` 슬라이스로 대체.

### 성능 (baseline vs 최적화, 동일 입력)

동일 머신에서 최적화 전 동결본(baseline)과 최적화본을 같은 씬·같은 캡처로 연속 측정. UV 스피어 씬, 정/후면 프로젝션 + masked/full 베이크 + postprocess 전체 파이프라인 기준.

| 설정 | baseline | 최적화 | 배속 | 출력 |
|---|---|---|---|---|
| 256px · 1 오브젝트 (4.6k tri) | 0.91s | 0.67s | **1.36x** | 바이트 동일 |
| 512px · 1 오브젝트 (9.2k tri) | 3.51s | 2.62s | **1.34x** | 바이트 동일 |
| 512px · 2 오브젝트 (18k tri) | 7.12s | 5.26s | **1.35x** | 바이트 동일 |
| 1024px · 2 오브젝트 (33k tri) | 27.31s | 19.64s | **1.39x** | 바이트 동일 |
| 1024px · 3 오브젝트 (49k tri) | 40.63s | 29.46s | **1.38x** | 바이트 동일 |
| 2048px · 1 오브젝트 (16k tri) | 50.9s | 37.0s | **1.37x** | 바이트 동일 |
| 2048px · 2 오브젝트 (33k tri) | 101.7s | 74.6s | **1.36x** | 바이트 동일 |
| 4096px · 1 오브젝트 (16k tri) | 381.2s | 268.7s | **1.42x** | 바이트 동일 |

베이크는 오브젝트마다 한 번씩 도므로 **오브젝트 수에 거의 선형**이다(2048에서 1→2 오브젝트가 정확히 2배). 실제 사용 크기 기준 오브젝트당 절약: 2048 ≈ 14초, 4096 ≈ 112초(≈1.9분). 고해상도일수록 텍셀 바운드 베이크 루프가 전체를 더 크게 차지해 최적화 이득이 조금 더 커진다(4096에서 1.42x).

단계별(512px·1오브젝트): 가장 무거운 **프로젝션 베이크 단계가 1.46x**(정면 0.68s→0.46s, 후면/스미어 1.64s→1.13s), 깊이·ID 버퍼 빌드 1.16x, postprocess 1.1x. 전체 병목인 텍셀 루프에 최적화가 집중돼 실제 베이크가 가장 크게 빨라진다.

### 검증 (기능 이상 없음)
- 단위/통합 테스트 전부 통과: `tests/test_core.py`(53개 체크), `tests/test_plugin_mock.py`, `tests/make_sphere_demo.py` 라운드트립.
- A/B 하니스로 baseline(최적화 전 동결본) vs 최적화본 출력 PNG 버퍼를 **바이트 단위 비교** — 다양한 고정 설정 + **랜덤 72시드**(원근/직교 카메라, 멀티머티리얼, 노멀 없는 메시, 각종 회전/스케일/피벗, 퇴화 삼각형, 여러 해상도/각도/블러)에서 전부 동일.
- 적대적 코드 리뷰 4개 관점(부동소수점 등가성 / 분기·인덱스 / 직교·엣지케이스 / 제거 코드 무결성) + 재현 검증에서 발산 0건.

### 알려진 향후 방향
- **더 큰 가속(선택)**: 바이트 동일 제약을 "허용오차 내 동일"로 완화하면 스캔라인 래스터화, 후면 포즈의 정점 단위 사전계산, 노멀 정규화 팩터링, 증분 무게중심 좌표 등으로 추가 가속이 가능하다(미세한 픽셀 차이 감수). 자세한 내용은 [docs/testing-and-known-issues.md](docs/testing-and-known-issues.md)의 "향후" 참고.
- 텍스처 4096 + 고폴리 + 다수 오브젝트는 여전히 무거우므로 1024~2048 권장.

## v1.0 — 최초 구현

Substance Painter의 Projection Paint를 정면/후면 2장으로 자동화한 리버스 프로젝션 베이커.

### 개념
1. 현재 카메라로 보이는 씬의 **정면을 캡처**(카메라는 돌리지 않음).
2. 대상 오브젝트를 월드 Y축으로 180도(턴테이블) 돌려 **후면을 캡처**한 뒤 원래 transform으로 복구.
3. 두 캡처를 각 메시의 UV로 리버스 프로젝션해 **메테리얼당 텍스처 PNG**를 굽는다.
4. 겹치거나 가려진 영역, 스치는 옆면은 처리 규칙에 따라 채우거나 마스킹.

### 구조
- **`CaptureFrontBackBake.py`** — Marmoset `mset` 접속부(UI + 오케스트레이션). API에 닿는 코드는 여기 한 곳에만.
- **`projbake/`** — `mset` 비의존 순수 파이썬 코어(맥에서도 단위 테스트 가능).
  - `linalg.py` 벡터/4x4 행렬/Euler→행렬/카메라 투영
  - `image.py` ImageRGBA 버퍼 + (알파 가중) 바이리니어 샘플링
  - `pngio.py` 표준 zlib만 쓰는 순수 파이썬 PNG 읽기/쓰기(필터 5종, 8/16bit, 팔레트 등)
  - `mesh.py` SceneMesh, 월드 변환(피벗 포함), 메테리얼 그룹핑, 삼각형 순회
  - `bake.py` 리버스 프로젝션 베이커(깊이+ID 버퍼 오클루전, 정/후면 분기, 옆면 마스킹)
  - `postprocess.py` 테두리 블러, UV 패딩, 투명 영역 채우기, 합성

### 핵심 기능
- 숨겨진 오브젝트(부모 체인 포함)·`invisibleToCamera` 제외
- **오브젝트별 격리 캡처**(나머지 숨김)로 앞 오브젝트에 가려 비던 뒷면까지 채움; 자기 겹침은 깊이 버퍼로, 오브젝트 간 번짐은 ID 버퍼로 차단
- Side Mask Angle 이상 기우는 면은 투명 마스킹, 테두리는 알파 가중 블러
- 메테리얼당 2장 출력: `_masked`(옆면 마스킹 + 블러 + RGB 패딩), `_full`(마스킹 없이 앞뒤 스미어 + 완전 불투명)
- 텍스처 프리셋(512~4096) + 커스텀 크기, 중간 캡처 파일 자동 정리
- 의존성 없음(Marmoset 내장 파이썬, 표준 라이브러리만)

### 설치
저장소 루트의 `install.ps1` / `install.bat`(Windows 원클릭) 또는 수동 복사. 자세한 내용은 [README](README.md) 참고.
